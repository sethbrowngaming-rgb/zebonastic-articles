How to Freeze a Player in Roblox Studio
— Movement, Camera, and Mouse

The simplest correct approach: what to change, why each line matters, and the traps beginners fall into.

I. The Code

Six lines. That's all it takes.

Place a LocalScript inside StarterPlayerScripts (or StarterCharacterScripts) and paste the following:

LocalScript — Freeze Movement & Camera, Free Mouse local Players = game:GetService("Players")
local UserInputService = game:GetService("UserInputService")

local player = Players.LocalPlayer
local character = player.Character or player.CharacterAdded:Wait()
local humanoid = character:WaitForChild("Humanoid")
local camera = workspace.CurrentCamera

-- Freeze movement
humanoid.WalkSpeed = 0
humanoid.JumpPower = 0

-- Lock the camera in place
camera.CameraType = Enum.CameraType.Scriptable

-- Let the mouse move freely
UserInputService.MouseBehavior = Enum.MouseBehavior.Default

That's the complete solution. The rest of this article explains exactly why each piece is written the way it is — so you can modify it confidently rather than copy-pasting blindly.

II. Line by Line

Nothing is accidental — not even the :Wait().

WalkSpeed = 0 and JumpPower = 0

The Humanoid object is what processes movement input. Setting WalkSpeed to zero means the engine still receives the player's input — the W/A/S/D keys still fire — but the resulting velocity is multiplied by zero, so nothing moves. Setting JumpPower to zero prevents the character from leaving the ground via the spacebar.

This is the right approach because it works with the engine rather than blocking input at the source. The character stays animated (idle state), physics still simulates, and you can restore movement instantly by setting these values back.

Deleting or disabling the Humanoid to freeze movement is a mistake. It breaks respawning, health, and animation state entirely — you'll get a corpse that can't be revived. Always set WalkSpeed and JumpPower to 0; never remove the Humanoid.

camera.CameraType = Enum.CameraType.Scriptable

Roblox cameras have several modes. The default is Custom, which lets the player orbit the camera around their character with the right mouse button or by dragging. Setting the type to Scriptable tells the engine: I will control this camera from code; don't respond to player input.

The camera stays exactly where it was when you made the change. It doesn't snap to a new position. If you want to also set its position, assign camera.CFrame after changing the type.

UserInputService.MouseBehavior = Enum.MouseBehavior.Default

By default in Roblox, the mouse cursor is hidden and locked to the center of the screen — this is what allows camera rotation to work. After locking the camera, you typically want the cursor visible and freely movable. MouseBehavior.Default unlocks it. The cursor appears and the player can move it anywhere on screen without affecting the camera.

Three separate systems, three separate fixes. Movement lives on the Humanoid. Camera lives on workspace.CurrentCamera. Mouse cursor lives on UserInputService. Change all three, or you'll freeze two and leave one still responding.

player.Character or player.CharacterAdded:Wait()

This line handles a timing issue. If the LocalScript runs before the character has loaded into the world, player.Character is nil — and the next line will error. The or player.CharacterAdded:Wait() part pauses the script until the character exists before continuing. It's a small thing, but it's the difference between code that works reliably and code that breaks on fast machines or after respawns.

III. Placement

Where you put the script changes when it runs.

LocationRuns WhenUse This If
StarterPlayerScripts Once per session, when the player joins You want to freeze the player immediately on join and keep it frozen across respawns
StarterCharacterScripts Every time the character spawns or respawns You want the freeze to re-apply after each death automatically
A Script in ServerScriptService On the server, for all players You want to freeze all players at once from the server side (requires a RemoteEvent to trigger the LocalScript)

For most use cases — a cutscene, a menu, a loading screen — StarterCharacterScripts is the right choice. It guarantees the Humanoid exists when the script runs, so the :Wait() call above is technically redundant there, but it doesn't hurt to keep it.

IV. Reversing It

Unfreezing is the exact same code, run backwards.

Unfreeze — Restore All Three Systems -- Restore movement
humanoid.WalkSpeed = 16 -- Roblox default
humanoid.JumpPower = 50 -- Roblox default

-- Restore camera control to the player
camera.CameraType = Enum.CameraType.Custom

-- Lock the mouse back to center (for camera orbiting)
UserInputService.MouseBehavior = Enum.MouseBehavior.LockCenter

The default WalkSpeed in Roblox is 16 and the default JumpPower is 50. If your game uses different values, store them in a variable before freezing so you can restore the correct numbers rather than guessing.

Setting camera.CameraType back to Custom without also restoring MouseBehavior to LockCenter will leave the player with a free cursor while the camera tries to orbit — the camera will feel broken. Always restore all three systems together.

Synthesis — What You Now Know

The three systems are independent. Movement (Humanoid), camera (workspace.CurrentCamera), and mouse cursor (UserInputService) don't share state. Changing one doesn't affect the others. This is why the fix requires exactly three changes, not one.

WalkSpeed = 0 is not the same as blocking input. Input still fires; the result is just zero velocity. This keeps the character in a valid state and makes it trivial to unfreeze.

Scriptable camera type is a full handoff. Once set, the engine won't touch the camera's position or rotation at all. You own it. If you don't set a CFrame, it stays exactly where it was — which is usually what you want for a freeze.
